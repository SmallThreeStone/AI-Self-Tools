# 产品文档索引

本目录收纳微信工具箱小程序相关的 **产品规划** 与 **产品优化** 两类文档，二者职责与版本规则不同，便于团队 **按路线图发版** 的同时 **小步优化体验**。

---

## 一、两类文档怎么区分

| 类型 | 目录 | 文件名模式 | 用途 |
|------|------|------------|------|
| **产品规划** | [`规划/`](./规划/) | `产品规划-v{主}.{次}.{修订}.md` | **里程碑路线图**：一大段时间内要交付的功能范围（如发现效率、备份增值、分享等）。 |
| **产品优化** | [`优化/`](./优化/) | `产品优化-v{主}.{次}.{修订}.md` | **体验与架构微调**：页面 IA、文案、交互收敛；可与规划并行或插在两次规划版本之间单独上线。 |

---

## 二、版本号怎么递进（建议）

### 产品规划（里程碑）

- 与你们对外沟通的 **小程序主版本** 对齐为佳，例如：`v1.0.2` → … → **`v1.3.0`**（游戏）→ **`v1.4.0`**（学单词 **总册**）→ 同线下子规划 **`v1.4.1`～`v1.4.3`** → 下一跨线大版本 **`产品规划-v1.5.0.md`**（待编写）。
- **同一 `v1.4.x` 大线下的拆册**：可用 **`产品规划-v1.4.1.md`、`v1.4.2.md`…** 表示子阶段（与 **`产品优化-v1.4.x`** 不同文件名前缀，**不共用同一含义的修订号**）。
- **新建文件**：**跨线大里程碑**（如 `v1.5.0`）与子规划拆册（如 `v1.4.1`）均 **单独一个 md**，便于评审与归档。

### 产品优化（与规划里程碑对齐 + 批次）

为避免「规划已到 1.2.0、优化还停留在 1.0.x」的错位感，建议：

| 段 | 含义 | 示例 |
|----|------|------|
| **主.次** | 与 **当前规划所处的里程碑线** 对齐——通常取 **仓库内最新一篇《产品规划》的主、次版本号**（例如里程碑已到 **`产品规划-v1.4.0.md`**，则新优化可用 **`1.4.x`**）。 |
| **第三位** | 在该里程碑线下的 **优化专题批次**：例如在 `1.2.x` 线下为 `v1.2.1`、`v1.2.2`。在 **`1.4.x`** 线下若已与 **`产品规划-v1.4.1`～`v1.4.3`** 子规划并存，建议优化文档从 **`v1.4.10`** 起编号，或采用 **`v1.4.4`、`v1.4.5`** 等与「规划子册修订号」错开的习惯，避免口头混淆。 |

**何时上调「主.次」**：当新增 **`产品规划-v1.5.0.md`** 等跨线里程碑时，下一篇优化可升为 **`v1.5.1`**（示例）。

- **发版说明**：仍建议在小程序更新日志里写一句「本期对应《产品优化-vx.x.x》」便于追溯。
- **暂缓课题**：可在文档标题或摘要标注「暂缓」，后续新开 **`产品优化-vx.x.x.md`** 接续。
