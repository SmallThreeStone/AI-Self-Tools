## 〇、为什么要单独写本文

团队在评估「百度 / 有道这类 **网址** 能不能长期用、有没有 **每天 N 次**」以及「若接 **正式接口** 要怎么 **付钱**」时，容易把三类不同概念混在一起：

1. **对方网站是否承诺了配额**（通常：**没有**）  
2. **我方小程序里自己做的上限**（有，且可配置）  
3. **若接开放平台 / 云厂商 TTS 的商务与计费**（有合同或控制台账单，与 1 无关）

本文把三条线拆开，并与仓库内 **学单词分包** 现有实现对应，便于产品、研发、商务对齐。

## 一、结论摘要（「次数限制」到底是谁的）

| 问题 | 产品结论 |
| ---- | -------- |
| 调用 `dict.youdao.com`、历史曾用的百度翻译类 **gettts** 等 **网页 / 非开放 API 形态** 的 URL，有没有面向你方小程序的、稳定公开的 **「每天 N 次」合同式配额说明**？ | **没有。** 这类能力本质是 **非正式 TTS 服务**：官方未对第三方小程序承诺 SLA、配额或长期行为；实际常见 **按 IP / 频率 / 风控临时限速**、返回 **错误页 / 非 MP3 体**、**改 URL 行为**，且 **随时可能变**。 |
| 那用户感觉「播多了会挂」是什么原因？ | 多为 **源站限流 / 反爬 / 网络与返回体变化**，不是你可依赖的「额度用完明天再来」式官方口径。 |
| 项目里「今日朗读次数已用完」从哪来？ | 来自 **产品侧 / 工程侧** 的 **本地每日计数与上限**（见 **§三**），**不是**有道/百度给你的「官方每日额度」。 |
| 若要 **稳定、可预期** 的整句朗读？ | 长期应接 **有计费与配额的正式服务**：如 **各云厂商语音合成开放 API**、**自建 TTS 网关**、或 **微信生态内带商务条款的能力（插件 / 服务市场 / 云开发绑定云账号等）** —— 具体选型由研发评估，本文 **§四** 只列典型付费维度与接入姿势。 |

## 三、当前工程实现（便于对照，非开发手册）

以下路径均在仓库 **`Wechat_Mini_Program/miniprogram/package-learn/`** 下。

| 模块 | 文件 | 产品含义 |
| ---- | ---- | -------- |
| **演示用 URL 策略** | `lib/tts.ts` | 注释写明为 **演示用在线 TTS**：主依赖 **有道 `dictvoice`**；例句用 **多 type、截断、词头** 多 URL 降级；**百度 gettts 不再作默认源**（部分环境 `downloadFile` 得到极小 HTML）。需在公众平台为 **`https://dict.youdao.com`** 配 **downloadFile** 合法域名。 |
| **链式下载与 MP3 校验** | `lib/learn-audio-play.ts` | 多 URL **依次** `wx.downloadFile`，校验 **真 MP3** 再播；失败打 `[learn-tts]` 日志，用户侧 Toast「演示朗读暂不可用…」类提示 —— 体现 **非正式源的不稳定性**。 |
| **本地每日配额** | `lib/learn-tts-quota.ts` | 存储键 **`learn_tts_quota_v1`**：**单词 + 例句** 成功起播合计计数；达上限 Toast「今日朗读次数已用完…」。 |
| **配额可配置** | `lib/learn-tts-config.ts` | 默认每日上限常量 **`LEARN_TTS_DEFAULT_DAILY_LIMIT`（当前为 80）**；存储键 **`learn_tts_config_v1`** 可覆盖 `dailyLimit`（`null` / `-1` 关闭上限等）。 |
| **例句缓存** | `lib/learn-tts-cache.ts` | 键 **`learn_tts_paths_v1`** 等；与 `tts.ts` 策略变更时 **缓存 scheme** 可递增，避免脏缓存。 |

**一句话**：链式 URL 是 **不接入正式 TTS 前提下的折中**；**每日 N 次** 主要是 **你们自己设的产品 / 成本护栏**，不是第三方「合同额度」。

## 四、若对接「正式」TTS：典型怎么付费、怎么接（口径级）

> **重要**：各云 **单价、免费试用额度、活动价** 经常调整。落地前请以 **厂商官网控制台 → 语音合成 / TTS → 定价** 为准；商务合同以 **订单 / 资源包** 为准。下表只描述 **常见模式**，不写承诺性数字。

### 4.1 通用原则（几乎适用于所有云开放 API）

| 步骤 | 说明 |
| ---- | ---- |
| **主体与实名** | 企业或个人在云平台 **注册并完成实名**。 |
| **开通产品** | 在控制台开通「语音合成」「长文本语音合成」等具体产品。 |
| **计费模型** | 常见为：**按字符数**（或按次数折算字符）、**资源包（预购）**、**后付费按日/按月账单**；部分音色 **单价更高**。 |
| **密钥安全** | **AppKey / Secret 不得写进小程序包**，须放在 **自有后端** 或 **云函数**，由小程序 **带用户身份** 请求你们服务，再由服务端调用 TTS。 |
| **合规** | 用户朗读文本是否出境、日志留存、未成年人场景等，按你们 **隐私政策** 与厂商 **数据处理条款** 评审。 |

### 4.2 常见厂商（产品选型时对比用）

| 类型 | 典型路径 | 付费大致长什么样 |
| ---- | -------- | ---------------- |
| **国内云：腾讯 / 阿里 / 百度 等** | 开放平台文档「语音合成」REST / SDK | **按万字符级** 或 **资源包** 为主；常有 **新客免费额度**（有有效期）；高阶音色 **加价**。需单独评估 **QPS / 并发** 是否另计费或限流。 |
| **有道智云等独立语音品牌** | 「有道开放平台」申请应用、选 **语音合成** 接口 | 多为 **调用量套餐** 或 **后付费**；以控制台为准。 |
| **微信生态内能力** | 例如 **同声传译插件**、**服务市场第三方语音插件**、或与 **云开发 + 云账号** 绑定的方案 | 可能同时存在 **微信侧授权费 / 插件年费** 与 **底层云厂商按量费**；**必须逐份阅读当前版本的插件说明与计费页**，不宜口头沿用旧帖数字。 |

### 4.3 与「当前 dictvoice 链式播放」相比，正式 API 多出来的成本项

| 项 | 演示 URL（现状） | 正式 API |
| -- | ---------------- | -------- |
| **音频费** | 无直接合同账单（风险转嫁为 **不稳定 + 合规灰区**） | **按量或资源包**，可预测。 |
| **研发与运维** | 客户端拼 URL + 下载链 | **服务端合成或签发短期 URL**、错误码、重试、监控。 |
| **域名** | 依赖第三方域名进小程序白名单 | 多为 **自有 API 域名** + 厂商 endpoint（服务端访问）。 |
| **音色与整句质量** | 受 **非 API 页面** 行为影响大 | 一般 **整句稳定**、可选音色，适合 v1.4.12 的「例句整句」体验目标。 |

## 五、编者建议：当前阶段怎么做？（直接回答）

### 5.1 结论：**当前阶段建议暂时保持现状**

在 **没有** 以下「必须升级」的触发条件之前，**建议继续沿用现有方案**：**有道 `dictvoice` 链式下载 + MP3 校验 + 本地每日上限（`learn_tts_quota_v1`）+ 本地缓存（`learn_tts_paths_v1`）**。  

理由用大白话说就是三条：

1. **已经能跑通产品**：用户能点读、有失败提示、有次数护栏，符合 [产品优化-v1.4.12.md](./产品优化-v1.4.12.md) 的交互定稿。  
2. **立刻接正式厂商 = 要花钱 + 要写服务端 + 要配域名与开票主体**，对「还在验证用户量 / 功能优先级」的阶段往往是 **性价比不高** 的一笔固定成本。  
3. **非正式网页没有「官方每天 N 次」**，但你们 **自己设的每日上限** 反而帮你们 **少碰风控、少背投诉峰值** —— 在现状下这是 **合理的产品手段**，不是「凑合」。

### 5.2 什么时候 **不该** 再拖、应启动「正式对接」（满足任一即可立项）

| 触发信号 | 含义 |
| -------- | ---- |
| **失败率明显升高** | 大量用户反馈「一直合成中 / 演示朗读不可用」，说明源站策略或网络侧已 **不可接受**。 |
| **规模或运营压力** | DAU 明显上升、推广活动在即，继续依赖非 API 源 **风险集中爆发**。 |
| **合规或品牌要求** | 法务 / 合作方要求 **可说明的音频来源与数据处理协议**（非正式 URL 很难写进对外说明）。 |
| **路线图依赖** | 已确定要做 [产品规划-v1.4.3.md](../规划/产品规划-v1.4.3.md) 类 **语音对话**、或 TTS 调用量将 **数量级上升**。 |

出现上表情况时：**不要**再在客户端叠更多「免费 URL」——应走 **§六**，选 **1 家正式 TTS + 自有后端网关**。

### 5.3 中长期路线（与 §四 对照）

| 阶段 | 建议 |
| ---- | ---- |
| **现在～触发前** | **保持现状**；把用户预期讲清楚（见下条）；监控失败类反馈与 `[learn-tts]` 日志。 |
| **触发后（中期）** | 选型 **1 家**正式云 TTS（英文学单词优先看 **英文整句质量、延迟、单价、文档成熟度**），按 **§六** 完成采购与对接；小程序 **只请求你们自己的 HTTPS 域名**；密钥 **只放在服务器**。 |
| **再往后（长期）** | 多场景共用同一 **TTS 网关**（单词、例句、对话），加 **成本看板** 与 **配额策略**（可与 v1.4.12 的「每日上限」合并治理）。 |

### 5.4 给运营 / 客服的一句「口径」（可自行改写）

> 「朗读由 **在线演示通道** 提供，受网络与服务策略影响，**不保证**与商用 TTS 同等稳定性；我们已限制每日次数以保障体验。」

（是否采用、是否加「法务审核」由你们自己定。）

## 六、给纯小白的：怎么采购、怎么对接正式 TTS（按顺序做）

下面假设你 **完全没有** 买过云服务、没有对接过 API。你只要 **按步骤往下勾**，把每一步交给「对应角色的人」去做即可。

### 6.0 先搞懂几个词（看到控制台就不懵）

| 词 | 白话 |
| -- | ---- |
| **实名认证** | 用身份证或营业执照，证明「这个云账号是谁的」。企业开发票、签协议通常要 **企业实名**。 |
| **主账号 / 子账号** | 主账号 = 老板付钱那个；子账号 = 给员工 **受限权限**（推荐：研发只有 **调用 TTS**，没有 **删库/提现**）。 |
| **AccessKey（AK）/ Secret（SK）** | 一对 **用户名+密码**，用来在 **服务器程序** 里调用云 API。**绝对不能**写进小程序前端代码或发到群里。 |
| **资源包 / 预付费** | 先 **买一笔字数额度**（例如「x 万字符」），用多少扣多少，**适合先控制预算**。 |
| **后付费** | 先开通，月底按实际用量 **出账单扣余额**；通常要满足厂商 **信用 / 余额** 条件。**适合**用量稳定后。 |
| **QPS / 并发** | 一秒钟能请求多少次合成。用户一多可能要 **提额**（有时单独计费或工单申请）。 |
| **服务端** | 可以是一台你们自己的服务器，也可以是 **微信云开发云函数**、阿里云函数计算等 —— 原则只有一个：**密钥放这里，不放小程序里**。 |

### 6.1 谁需要参与（最小班子）

| 角色 | 要做什么 |
| ---- | -------- |
| **你（负责人 / 产品）** | 选 **1 家**厂商、拍板 **预算上限**、确认 **要不要企业发票**。 |
| **财务或行政** | 提供 **营业执照**、**对公打款 / 绑卡充值**、后续 **收发票**。 |
| **研发** | 注册控制台、开通产品、写 **后端接口**、改小程序 **只调自家域名**、联调与上线。 |

个人主体也能买很多云服务，但 **企业发票、合同** 一般走企业实名更方便。

### 6.2 采购与开通：推荐顺序（勾一项做一项）

**第 1 步 —— 选定一家（先选一家就行）**  
在 **腾讯云 / 阿里云 / 百度云 / 有道智云** 里先选 **一家**（不必一次买四家）。选型时可让研发填一张很简单的表：**英文整句试听、文档链接、是否有「长文本」、预估每月多少万字、是否已有同厂其他产品**。

**第 2 步 —— 注册账号并完成实名**  
打开厂商官网 → 注册 → 按提示上传 **身份证** 或 **营业执照**。未完成实名，通常 **无法购买** 或无法开票。

**第 3 步 —— 进入控制台，搜索「语音合成」或「TTS」**  
在控制台 **产品搜索框** 里输入上述关键词 → 进入产品页 → 点 **立即开通 / 开通服务**（不同厂商按钮名字略不同）。

**第 4 步 —— 打开「定价 / 计费说明」，决定先买资源包还是先后付**  

- **心里没底、怕超支**：优先 **资源包** 或 **小额充值 + 设置余额告警**。  
- **用量已可估算**：可与财务商量 **后付费 + 每月预算上限**（再在控制台设 **用量告警**）。

**具体多少钱**：必须以控制台 **当前定价页** 为准；让研发或财务把「**1 个用户每天点 30 次、每次平均 80 个英文字符**」这种 **粗算表** 做一页 Excel 就够决策用，不必你手算公式。

**第 5 步 —— 创建访问密钥（只在服务器用）**  
在控制台的 **「访问管理 / API 密钥」** 一类菜单里创建 **AK/SK**。  

- **复制 SK 时只显示一次**，立刻存进公司的 **密码管理器** 或交给研发写入 **服务器环境变量**，不要发微信明文。  
- 建议同时建 **子账号 + 仅 TTS 权限** 的密钥，降低泄露损失面。

**第 6 步 —— 研发做「中间层」接口（这是对接的核心）**  

小程序 **不要** 直接携带密钥去调腾讯云 / 有道。标准做法是：

1. 你们有一个 **HTTPS 域名**（例如 `https://api.你们公司.com/tts`）。  
2. 小程序只请求这个地址，带上 **业务参数**（例如「要朗读的句子 id」或「脱敏后的文本」——具体由研发按隐私评审定）。  
3. **服务器**用 AK/SK 调云厂商 TTS → 拿到 **音频文件或短时有效的下载 URL** → 返回给小程序播放。

这样即使有人反编译小程序，也 **偷不走** 你的云厂商密钥。

**第 7 步 —— 微信小程序后台配置「合法域名」**  

1. 登录 [微信公众平台](https://mp.weixin.qq.com/) → 你的小程序。  
2. **开发 → 开发管理 → 开发设置 → 服务器域名**。  
3. 把 **第 6 步** 的 `request` 域名（以及若直接返回音频 URL 且走 `downloadFile`，则还要配 **downloadFile 合法域名**）按微信要求填进去。  
4. **保存**；域名需 **备案** 等要求以微信当时规则为准。

**第 8 步 —— 联调、灰度、再全量**  

- 先 **开发者工具 + 真机** 各测一轮：弱网、失败重试、每日上限与云端是否一致。  
- 再 **小比例用户** 或 **仅内测白名单** 打开新链路，观察 **错误率与账单**。  
- 最后再全量；旧 `dictvoice` 链可保留为 **降级**（可选，由研发评估维护成本）。

**第 9 步 —— 发票与合同（财务友好版）**  

- 控制台 **费用中心** 一般可申请 **电子发票**；企业抬头以财务要求为准。  
- 若要走 **纸质合同 / 框架合同**，找厂商 **客户经理** 或控制台里的 **商务咨询** 入口（各厂名字不同）。  
- **对公汇款**路径也在费用中心 / 充值页，按财务制度走即可。

### 6.3 若走「微信生态里的插件 / 服务市场」（与纯云略有不同）

大致仍会经历：**阅读插件计费说明 → 在小程序后台添加插件 → 绑定或跳转云厂商账号扣费 → 研发按插件文档接入**。  

因为 **插件条款与价格会改版**，这里不写死步骤编号；原则与 **§六.2 第 6 步** 相同：**钱从哪扣、密钥在哪、域名怎么配**，三项必须当面问清研发并留下 **截图或链接归档**。

### 6.4 常见坑（小白提前避雷）

| 坑 | 怎么办 |
| -- | ---- |
| 把 **Secret 写进小程序** | **禁止**；泄露后别人刷你的额度。 |
| 只买了资源包 **没设告警** | 在控制台设 **余额 / 用量告警** 到负责人手机或邮件。 |
| **没备案的域名** 无法配置 | 提前问研发域名与备案进度。 |
| 以为「买了云就自动能播」 | 必须 **研发写接口 + 小程序改调用** 才能接上。 |
