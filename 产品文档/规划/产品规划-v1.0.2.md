## 二、基线能力盘点（与研发对齐）

以下为 v1.0.1 已具备的核心资产，后续规划需**复用而非推倒**：

- **架构**：`tools.ts` 注册表、首页 / 分类 / 我的、搜索与别名、收藏 / 最近使用 / 猜你喜欢、明暗主题。
- **分包**：`package-tools`（逻辑类工具）、`package-canvas`（画布 / 相册类）。
- **工具覆盖面（节选）**：决策转盘、记分牌、真心话大冒险、手持弹幕、亲戚称呼、分账、掷硬币、番茄钟、折扣 / 单位换算 / 密码 / BMI、二维码、倒数日、补光灯、直尺、体脂与代谢估算、调色板、电子木鱼，以及画布类（白板、宫格切图、水印、文字转图）等。

**结论**：v1.0.2 不适合盲目加大量新工具，应优先 **体验闭环 + 数据可信说明 + 审核友好**，再规划 1.1.0 的功能增量。

## 三、用户与场景假设

| 人群 | 典型场景 | 产品诉求 |
|------|----------|----------|
| **高频本地工具用户** | 换算、计时、生成二维码、临时记分 | 打开快、路径短、状态可恢复 |
| **聚会 / 社交轻场景** | 转盘、真心话、弹幕、分账 | 上手零说明、弱网可用 |
| **颜值与创作向** | 水印、切图、配色 | 导出稳定、权限说明清晰 |

## 四、版本路线图（建议）

### 4.1 v1.0.2 —— 「稳定与可信」小版本

**目标**：降低差评与审核风险，提升「愿意再次打开」的概率。

| 优先级 | 模块 | 方向 | 验收要点 |
|--------|------|------|----------|
| P0 | **全局** | 崩溃与分包加载失败兜底（重试 / 提示文案） | 弱网下进入工具页可理解失败原因 |
| P0 | **我的 / 关于** | 版本号展示与更新说明入口（链接至公众号 / 文档或页内 dev-log） | 用户可知当前是否为最新逻辑 |
| P1 | **健康 / 测量类**（BMI、体脂代谢、直尺） | 统一「免责声明」组件：估算值、非医疗 / 非计量器具 | 文案一致、可一键关闭 |
| P1 | **权限** | `requiredScopes` 与首次能力调用前说明对齐（剪贴板、相册） | 与隐私协议 / 用户预期一致 |
| P2 | **搜索** | 热搜词 / 最近搜索（仅存本地） | 降低「找不到工具」的挫败感 |

**不做清单（防止范围蔓延）**：不接登录、不上云同步、不大改 Tab 结构。

---

### 4.2 v1.1.0 —— 「发现效率 + 工具增量」

**目标**：让已有工具被用起来，并可控地扩展 SKU。

| 方向 | 说明 | 价值假设 |
|------|------|----------|
| **首页重构（轻量）** | 「常用工具」置顶编辑、分类快捷入口 | 减少三步以上的跳转 |
| **工具详情或长按菜单** | 展示标签、权限说明、最近一次使用时间 | 提升信任与可发现性 |
| **2～4 个新工具（择优）** | 例如：简易备忘录（纯本地）、倒计时铃声预设、色盲友好对比检查 | 每条需对应明确场景与数据键规范 |
| **埋点（若合规允许）** | 仅统计页面访问与工具打开次数（匿名） | 用数据验证下一批工具优先级 |

---

### 4.3 v1.2.0 —— 「可选增值」（视资源而定）

| 方向 | 说明 |
|------|------|
| **配置导出 / 导入** | 扩展现有「数据导出」：收藏 + 部分工具本地数据 JSON |
| **深色模式跟随系统** | `wx.onThemeChange` 与手动切换并存 |
| **分享卡片优化** | 指定页面 `onShareAppMessage` 差异化标题（工具页 > 默认首页） |

## 六、成功指标（建议）

在匿名且不采集个人信息前提下，可用 **微信后台「数据分析」** 结合简易表格手动记录：

| 指标 | 含义 | 1.0.2 期望 |
|------|------|------------|
| **次日留存（可选）** | 整体粘性参考 | 不低于发版前基线，不因改版明显下降 |
| **人均打开工具次数** | 是否「用得起来」 | 环比持平或略升 |
| **分包页面报错率（主观）** | 用户反馈「打不开」类 | 明显下降 |

（若暂未接入自定义埋点，则以 **反馈量减少 + 主观可用性测试** 替代。）

## 七、风险与依赖

| 风险 | 应对 |
|------|------|
| **工具数量增多导致分包体积与审核时长** | 严格控制单工具资源；画布类统一压缩策略 |
| **「医疗 / 测量」表述触发审核** | 统一免责声明；避免「治疗」「诊断」等用语 |
| **后台计时、闹钟类预期** | 产品文案明确「受系统限制」，避免过度承诺 |

---

## 八、开放问题（需产品 / 研发共同决策）

1. **目标用户**更偏「自用效率」还是「聚会玩乐」？—— 决定首页默认推荐算法权重。  
2. **是否长期坚持零账号**？—— 若坚持，所有「云」能力只能通过导出文件由用户自行备份。  
3. **后续是否开放 UGC（用户自定义题库等）**？—— 涉及审核与内容安全，建议 1.1.0 后再评估。
